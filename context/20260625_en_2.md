**Exercise 1: Can the following code compile?**

```rust
fn main() {
    let s = String::from("hello");
    let s_ref = &s;
    let s2 = s;
    println!("{}", s_ref);
}
```

**Answer: No, it cannot compile.** The error message will say: `cannot move out of 's' because it is borrowed`.

**In-depth analysis:** When `let s_ref = &s` is executed, Rust creates an **immutable borrow** of `s`. This borrow lives from the point of creation until the last use of `s_ref` (here, in `println!`). While the borrow is still active, `let s2 = s` attempts to **move** ownership of `s` to `s2`. This causes two serious problems:
1. After `s` becomes invalid, `s_ref` would become a **dangling pointer** pointing to invalid memory.
2. It violates the ownership rule that "the original value cannot be moved while borrowed."

Rust's borrow checker catches this at compile time – it enforces the rule that **when a value is borrowed, its ownership cannot be moved or dropped until the borrow ends**. This guarantees that references always point to valid data.

**Exercise 2: Implement `first_word(s: &String) -> &str`**

This is a classic string slice exercise. A recommended implementation:

```rust
fn first_word(s: &String) -> &str {
    let bytes = s.as_bytes();          // convert String to byte array
    for (i, &item) in bytes.iter().enumerate() {
        if item == b' ' {              // found a space, return slice up to this point
            return &s[..i];
        }
    }
    &s[..]                             // no space found, return the whole string
}

fn main() {
    let s = String::from("hello world");
    let word = first_word(&s);
    println!("First word: {}", word);   // prints "hello"
    
    // Note: if we try to modify s here (e.g., s.clear()), it will fail to compile
    // because word still borrows s.
}
```

**Key design intention**: The return type is `&str` (a string slice), which is essentially an **immutable borrow** pointing to a contiguous sequence of bytes inside `s`. This means that as long as `word` exists, `s` cannot be modified or dropped – the borrow checker enforces this throughout the remaining scope of `main`.

### 1. Review: The "Inconvenience" of Ownership

In the previous article, we learned about the ownership rules:

```rust
fn main() {
    let s1 = String::from("hello");
    let s2 = s1;           // move, s1 becomes invalid
    // What if we want to keep using s1?
}
```

If every time we pass a value to a function or assign it to another variable we transfer ownership, the code becomes extremely cumbersome – you would constantly have to "pass back" ownership via return values:

```rust
fn main() {
    let s1 = String::from("hello");
    let (s2, len) = calculate_length(s1);  // s1 moved in, returned via tuple
    println!("'{}' length is {}", s2, len);
}

fn calculate_length(s: String) -> (String, usize) {
    let length = s.len();
    (s, length)   // return ownership back
}
```

This is safe but has zero ergonomics. **Borrowing** was designed to solve this – it allows you to **temporarily use a value without taking ownership**, just like borrowing a book from the library without buying it.

### 2. References and Borrowing

A **reference** is a pointer that points to a value but does not own it. The operation of creating a reference is called **borrowing**.

#### 2.1 Immutable References (`&T`)

```rust
fn main() {
    let s1 = String::from("hello");
    let len = calculate_length(&s1);   // pass &s1, an immutable reference
    println!("'{}' length is {}", s1, len);  // s1 is still valid!
}

fn calculate_length(s: &String) -> usize {  // s is a reference to a String
    s.len()                                 // can read, but cannot modify
} // s goes out of scope, but because it has no ownership, nothing is dropped
```

Key semantics:
- `&s1` creates a reference to `s1` but **does not own** it.
- The parameter `s` in `calculate_length` is a reference; the function can only read, not modify.
- After the call, `s1` remains intact with full ownership.

#### 2.2 Mutable References (`&mut T`)

To modify a borrowed value, we need a **mutable reference**:

```rust
fn main() {
    let mut s = String::from("hello");   // must be mut
    change(&mut s);                      // pass a mutable reference
    println!("{}", s);                   // prints "hello, world"
}

fn change(s: &mut String) {
    s.push_str(", world");   // modify the original via mutable reference
}
```

But mutable references come with a **very strict restriction**:

> **In a given scope, you may have either one mutable reference to a piece of data, or any number of immutable references, but not both at the same time.**

```rust
fn main() {
    let mut s = String::from("hello");
    
    let r1 = &mut s;
    let r2 = &mut s;  // compilation error! cannot have two mutable references
    
    println!("{}, {}", r1, r2);
}
```

```rust
fn main() {
    let mut s = String::from("hello");
    let r1 = &s;      // immutable reference
    let r2 = &s;      // multiple immutable references are allowed
    let r3 = &mut s;  // compilation error! cannot have immutable and mutable references together
    
    println!("{}, {}, {}", r1, r2, r3);
}
```

**Why so strict?** This rule eliminates **data races** at compile time – a data race occurs when two pointers access the same memory simultaneously, with at least one writing, causing undefined behavior. In C/C++ this is a runtime nightmare, while in Rust it's a compile-time error.

**Note**: A reference's scope ends at its **last use**, not at the end of the lexical scope. Therefore, the following is legal:

```rust
fn main() {
    let mut s = String::from("hello");
    let r1 = &s;
    let r2 = &s;
    println!("{} and {}", r1, r2);  // r1, r2 last used here
    // after this, r1 and r2 are no longer considered alive
    
    let r3 = &mut s;               // OK! because immutable references are no longer in use
    r3.push_str(" world");
    println!("{}", r3);
}
```

This subtlety is called **NLL (Non-Lexical Lifetimes)** , an important improvement introduced in Rust 2018 that makes the borrow checker more intelligent.

### 3. Dangling References and the Birth of Lifetimes

In C/C++, one of the most painful problems is **dangling pointers** – pointers that point to memory that has already been freed. Rust eliminates this entirely at compile time.

The following code **cannot compile**:

```rust
fn main() {
    let reference_to_nothing = dangle();
}

fn dangle() -> &String {      // returns a reference to a String
    let s = String::from("hello");
    &s                         // reference points to s, but s is about to be dropped
} // s goes out of scope, memory is freed, reference becomes dangling
```

The compiler will complain: `missing lifetime specifier`, and hint that `this function's return type contains a borrowed value, but there is no value for it to be borrowed from`.

How does Rust prevent this? The answer is **Lifetimes** – a static analysis mechanism the Rust compiler uses to track the validity range of all references.

---

### 4. Lifetime Annotations

In most cases, Rust can **automatically infer** lifetimes (via the "lifetime elision rules"), so we don't need to annotate them manually. However, in some scenarios where the compiler cannot determine the relationship between the lifetimes of references, we must explicitly annotate.

Lifetime annotation syntax is distinctive: it starts with a `'` (apostrophe) followed by a lowercase name, like `'a` or `'b`. It **does not change the actual lifetime** of a reference; it merely describes the **relationship** between the lifetimes of multiple references.

#### 4.1 The Classic Example: Returning the Longer of Two References

```rust
fn longest<'a>(x: &'a str, y: &'a str) -> &'a str {
    if x.len() > y.len() { x } else { y }
}

fn main() {
    let string1 = String::from("abcd");
    let string2 = "xyz";   // string literals are of type &'static str
    
    let result = longest(&string1, string2);
    println!("The longer string is {}", result);
}
```

The `<'a>` is a generic lifetime parameter, indicating that `x`, `y`, and the return value **all have the same lifetime `'a`**. More precisely, it means: **the returned reference will live no longer than the shorter of the lifetimes of `x` and `y`**.

Let's look at an erroneous example to understand this:

```rust
fn main() {
    let string1 = String::from("long long");
    let result;
    {
        let string2 = String::from("xyz");
        result = longest(&string1, &string2);
    } // string2 is dropped here
    println!("{}", result);  // compilation error! result might refer to freed string2
}
```

The compiler will point out that `result` requires the lifetime `'a` to extend at least to the `println!`, but `string2`'s lifetime ends at the inner scope – Rust prevents this dangling reference through lifetime checking.

#### 4.2 Lifetimes in Structs

When a struct holds references, lifetimes must be annotated:

```rust
struct ImportantExcerpt<'a> {
    part: &'a str,   // means this reference must not outlive the struct instance
}

fn main() {
    let novel = String::from("Call me Ishmael. Some years ago...");
    let first_sentence = novel.split('.').next().expect("Could not find a period");
    let excerpt = ImportantExcerpt {
        part: first_sentence,
    };
    // excerpt's lifetime is tied to first_sentence
    println!("{}", excerpt.part);
}
```

Here `ImportantExcerpt<'a>` tells the compiler that the struct instance **cannot outlive** the data referenced by the `part` field. If `novel` were dropped before `excerpt`, it would be a compile error.

#### 4.3 The Static Lifetime `'static`

`'static` is a special lifetime that indicates the reference is valid for **the entire duration of the program**. All string literals (e.g., `"hello"`) have the `'static` lifetime because they are embedded directly in the read‑only section of the binary.

```rust
let s: &'static str = "I exist for the entire program lifetime";
```

But be cautious: **do not treat `'static` as a "silver bullet"**. In many cases, marking a reference with `'static` overly restricts the usage of a function, when you really only mean "this reference must live at least this long." In most scenarios, generic lifetime parameters are a more flexible choice.

### 5. The Joint Rules of Borrowing and Lifetimes (Three Golden Rules)

Combining ownership, borrowing, and lifetimes, the core of Rust's memory safety can be summarised as three compile‑time rules:

1. **Ownership rule**: Every value has exactly one owner; when the owner goes out of scope, the value is dropped.
2. **Borrowing rules (reference rules)** :
   - At any given time, you may have either **one** mutable reference, or **any number** of immutable references, but not both.
   - All references must always be valid (i.e., no dangling references).
3. **Lifetime rule**: The lifetime of a borrower (the user of a reference) must not exceed the lifetime of the borrowed data (the owner of the value).

These three rules constitute a complete static measurement system for memory safety in Rust – it requires no runtime GC, nor manual memory management by the developer; all safety checks are completed at compile time.

### 6. In‑Depth Practical Example: A Safe "Word Splitter"

Let's combine mutable references, slices, and lifetimes to build a practical utility function.

**Requirement**: Implement a function that accepts a string slice and returns a structure with an iterator‑like interface that returns words one by one.

```rust
struct WordSplitter<'a> {
    text: &'a str,
    position: usize,
}

impl<'a> WordSplitter<'a> {
    fn new(text: &'a str) -> Self {
        WordSplitter { text, position: 0 }
    }
    
    fn next_word(&mut self) -> Option<&'a str> {
        let bytes = self.text.as_bytes();
        let len = bytes.len();
        
        // skip leading whitespace
        while self.position < len && bytes[self.position] == b' ' {
            self.position += 1;
        }
        
        if self.position >= len {
            return None;
        }
        
        let start = self.position;
        while self.position < len && bytes[self.position] != b' ' {
            self.position += 1;
        }
        
        Some(&self.text[start..self.position])
    }
}

fn main() {
    let sentence = String::from("   Rust lifetimes   in   practice  ");
    let mut splitter = WordSplitter::new(&sentence);
    
    while let Some(word) = splitter.next_word() {
        println!("Word: '{}'", word);
    }
    // Output:
    // Word: 'Rust'
    // Word: 'lifetimes'
    // Word: 'in'
    // Word: 'practice'
}
```

**Lifetime analysis**:
- The `WordSplitter<'a>` struct holds `text: &'a str`, meaning the struct instance must live shorter than the string data pointed to by `text`.
- `next_word()` returns `Option<&'a str>`, guaranteeing that the returned string slices have the same lifetime as `text` – they point into the interior of `text` and are valid as long as `text` is valid.
- In `main`, `sentence` is a `String`; we create a reference via `&sentence` and pass it in. This reference's lifetime covers the entire use of `splitter`, so it is completely safe.

### 7. Summary

Today we crossed one of the most critical thresholds in learning Rust:

- **Two kinds of references**: immutable `&T` and mutable `&mut T`, and their concurrency‑safety rules.
- **Core constraints of borrowing**: the original value cannot be moved or dropped while borrowed; mutual exclusion rules between multiple borrows.
- **The essence of lifetimes**: not a runtime mechanism, but a compile‑time "annotation language" used to verify reference validity.
- **Lifetime annotation syntax**: generic `'a` parameters, lifetimes in structs, and the static `'static` constant lifetime.
- **Practical implementation**: safely building an iterator‑style struct that borrows internal data using lifetimes.

If ownership is the "constitution" of Rust's memory safety, borrowing and lifetimes are its "judicial interpretations" – together they form a memory safety system with zero runtime overhead. For readers with a computer science background, you can think of it as: **Rust encodes resource lifetime constraints into the type system, so that illegal memory operations are rejected at compile time by the type checker**. This is the most powerful expression of type‑driven programming in a systems language.
