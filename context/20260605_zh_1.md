\# AI Agent 核心概念深度解析：从 Model、Tool、Skill 到 Harness 工程



> 当模型能力趋于饱和，真正的壁垒正在模型之外——那些让智能体“被放心地做”的外围基础设施。



\## 引言：超越模型的时代



2026年2月，OpenAI发布了一篇题为《Harness Engineering：Leveraging Codex in an Agent-First World》的文章。文中引用了LangChain做过的一个实验：不改模型，不改Prompt技巧，不加Few-shot，仅仅改造Harness，就把Terminal-Bench 2.0上的准确率从52.8%提到了66.5%。



这个数字震撼了很多从业者：模型的性能瓶颈，越来越不在模型本身。



如果你还在纠结“GPT-5.4比Claude 4.5强多少”，或许你已经不在AI Agent落地的一线了。2026年5月，这个问题已经失去了意义——头部模型的差距在快速收窄，MMLU、HumanEval这些基准上前五名的分差已不到两个点。同时，另一个现象正在变得要命：同样用Claude 4.5，有人搭建的Agent能自动处理工单、修Bug、跑完整条CI流水线；有人只跑出了一个“帮我写段代码”的聊天窗口。



差距在哪里？不是在模型能力，而是在模型之外——在Tool的丰富度、Skill的生态深度和Harness工程的系统成熟度。



2026年4月，上海交通大学联合中山大学、上海创智学院、卡内基梅隆大学及OPPO等机构，在arXiv提交了一篇54页的综述论文，首次以 \*\*“外部化”\*\* 为统一视角，系统梳理了LLM Agent的记忆、技能、协议与Harness工程\*\*四大支柱\*\*。论文的核心观点令人深思：Agent的实际进展，越来越取决于模型之外的外部认知基础设施，而非模型本身的能力提升。



本文将沿着这一外部化的思想脉络，系统拆解AI Agent的四个核心概念——Model（模型）、Tool（工具）、Skill（技能）和Harness（驾驭工程），试图为读者提供一个从底层原理到工程实践、从概念辨析到系统设计的完整认知框架。





\## 第一章：Model——智能体的认知核心



\### 1.1 模型：从“大脑”到组件



任何一个AI Agent系统，都是从模型开始的。模型是Agent的“智能发动机”——负责理解用户意图、进行推理判断、生成工具调用决策和最终响应。但如果只把Agent理解成“大模型+几行提示词”，就远远低估了这个系统的复杂性。



在Agent的四层架构中，Model处于最核心的决策层，但它既不直接感知外部世界，也不直接执行操作。它的全部输入来自Harness编排好的上下文和对话历史，它的全部输出要么是自然语言响应，要么是结构化工具调用请求。这种“隔离”不是模型的局限，而是架构设计的刻意为之——模型只负责它最擅长的事情：理解和生成。



\### 1.2 模型能力的演进：从“参数为王”到“外部扩展”



上海交大综述论文将Agent能力的载体演进分为三个阶段，清晰地揭示了模型角色如何被逐步外移：



\*\*阶段一：权重层（2022-2023）\*\* ——能力几乎等同于模型参数，扩展定律主导叙事。在这一阶段，模型能力被等同于参数规模：模型越大、参数越多，表现就越好。但这种方式的问题很快暴露：知识难以选择性更新，行为难以审计，个性化几乎无从实现。整个系统的可靠性完全依赖于模型的“内部记忆”，而模型既无法记住跨会话的状态，也无法动态更新知识。



\*\*阶段二：上下文层（2023-2024）\*\* ——Prompt工程、思维链、RAG兴起，模型保持冻结，提示模板快速迭代。在这一阶段，开发者发现了一个重要的“表征变换”：通过将外部知识注入上下文，可以将困难的“回忆”问题转化为更可靠的“识别”问题。上下文窗口的扩展使得RAG成为主流范式，但上下文窗口本身有限且短暂，跨步骤的协调始终脆弱。



\*\*阶段三：Harness层（2024至今）\*\* ——关注重心从参数知识和提示工程，逐步外移到Harness层级的基础设施。模型继续演进，但真正决定系统能力的，不再是模型本身有多强，而是它被什么样的外围系统包裹。



这种演进揭示了一个深刻的变化：\*\*模型正在从系统的“主角”变成系统的“组件”\*\* 。就像在传统软件工程中，数据库是重要组件但不是系统本身，模型在Agent架构中的定位越来越清晰——它是认知引擎，但不是整个智能体。剩下的所有工作——如何持久化记忆、如何调用工具、如何编排任务、如何保证安全可靠——都落在了Harness的肩上。



\### 1.3 当前Agent对模型的核心需求



随着Agent从单轮问答向多步骤任务执行演进，开发者对模型能力提出了全新的三大核心需求：



\*\*任务自主编排能力\*\*——从单轮对话到多步骤任务链。模型需要能够理解复杂目标并将其分解为可执行的子任务序列，这不仅需要强大的推理能力，还需要对任务执行的中间状态有持续的感知能力。



\*\*跨领域工具调用能力\*\*——从文本生成到系统级操作。模型需要能够理解何时调用工具、选择什么工具、生成什么参数，并在多轮工具调用中保持上下文连贯。



\*\*复杂任务持续优化能力\*\*——从静态输出到动态迭代。模型需要能够从失败中学习、根据执行反馈调整策略、在长周期任务中保持目标一致性。



这些需求恰恰解释了为什么模型本身不再是Agent的唯一决定因素——这些能力中的很大一部分，需要由外部的Harness来辅助实现。



\### 1.4 模型并非万能的：三个结构性错配



上海交大的综述论文指出，单纯依赖模型作为Agent核心面临着三个结构性错配：



\*\*连续性错配\*\*——上下文窗口有限且短暂，模型无法跨会话稳定保持状态。每次会话都是全新开始，之前积累的上下文需要从头重建。



\*\*一致性错配\*\*——复杂的多步骤流程往往被重新推导，而不是被稳定执行。同样的任务，在不同时机调用，执行路径和质量难以保证一致。



\*\*协调性错配\*\*——与工具、服务和其他Agent的交互依赖临时约定，脆弱且不可移植。接口一旦变动，整条调用链路可能同步失效。



模型很强大，但它天生不具备跨会话记忆、不具备代码执行环境、不具备工具调用的实际能力、不具备安全约束与权限控制。这些缺陷并非模型设计失误，而是其架构定位的必然结果。因此，我们需要的不是“更强的模型”，而是让现有模型能够被安全、稳定、高效地使用的系统化方法。这就是Tool、Skill和Harness工程登场的根本原因。





\## 第二章：Tool——连接模型的“手脚”



\### 2.1 从“会思考”到“能做事”的关键一跃



早期的大语言模型就像一位“知识渊博但手脚受限”的顾问——它们能回答问题、编写代码、生成文案，但一旦涉及获取实时信息、操作外部系统或执行多步骤任务，就会显得力不从心。根本原因在于：纯文本模型只能依赖训练数据来“认知”世界，无法触达实时信息，更无法直接作用于外部世界。



Tool Use（工具使用）正是打破这一限制的关键机制。通过工具调用，Agent能够查实时天气、访问企业数据库、执行代码、调用业务API、操作外部系统——这也是当前大厂AI岗位重点考察的核心能力之一。



\### 2.2 Tool Use的核心本质：模型决策 + 程序执行



很多初次接触Tool Use的开发者会陷入一个误区：误以为模型本身“能直接调用接口”“能独立执行代码”“能操控操作系统”。但2026年主流Agent的工具调用链路，核心逻辑只有一句话：



\*\*模型负责判断、选择和生成工具调用请求；应用程序负责真正执行操作；执行结果回传给模型后，由模型继续推理或组织最终回答。\*\*



工具调用的本质从来不是“模型自己动手”，而是模型输出一份结构化的动作指令，外部程序替它把指令落地执行。无论是OpenAI 2026年最新的Function Calling V3版本，还是Anthropic Claude 4的Agent机制，都将这个闭环明确拆解为五步：



1\. 给模型配置工具（通过JSON Schema定义）

2\. 模型判断是否需要调用工具，输出工具调用请求

3\. 应用侧代码执行工具/API调用

4\. 将工具执行结果回传模型

5\. 模型基于新信息输出最终回答或继续调用工具



这个循环持续到模型决定结束。真正的Agent，不是“有工具列表”，而是“有工具调用闭环”。Tool Use不是单次API技巧，而是一种运行时机制。



\### 2.3 工具定义：把API“翻译”成模型能理解的语言



模型之所以能调用工具，不是因为它天生认识你的内部API，而是因为你先把工具“描述”给了它。这一步通常要提供三类信息：



\- \*\*工具名\*\*：唯一标识符，用于模型在输出中引用

\- \*\*工具说明\*\*：详细描述工具的功能和使用时机——这往往比参数Schema更重要

\- \*\*输入参数Schema\*\*：以JSON Schema格式定义的参数结构，包括参数名、类型、是否必填等



Anthropic文档专门强调，工具描述应尽可能详细。模型越清楚“这个工具是干什么的、该怎么用”，调用效果就越好。背后的道理很简单：工具定义本质上是把程序员的领域知识翻译给模型，翻译得越准确，模型的决策就越可靠。



\### 2.4 主流平台的工具调用机制对比



目前主流的大模型平台——OpenAI、Google Gemini、Anthropic Claude——都提供了各自的工具调用机制，但它们在设计理念、执行流程和控制粒度上有明显差异。



\*\*OpenAI Function Calling\*\*是最早普及的机制。其核心思想是让模型不直接“编造答案”，而是判断什么时候该调用函数，并输出结构化参数，由开发者代码执行函数。模型输出类似`{"name": "get\_weather", "arguments": {"city": "Beijing"}}`的JSON结构，格式稳定、易于解析，适合工业级集成。



\*\*Google Gemini Function Calling\*\*在OpenAI的基础上更强调自动化编排。Gemini更偏向“Agent Native”设计，一次任务中可以自动调用工具A、再调用工具B、然后继续推理，多步任务更加自然流畅。



\*\*Anthropic Claude Tool Use\*\*则在安全性和可控性上发力。Claude的输出是高度规范化的tool\_use block，包含type、name和input字段，强调清晰的工具边界和可审计性，适合对安全要求严格的场景。



\### 2.5 工具的智能调度与组合



在真实生产环境中，单次工具调用远远不够。一个复杂任务往往需要多工具协同、多轮迭代。例如，一个让LLM控制电脑浏览器的场景，需要把“鼠标移动”“点击”“滚动”“截图分析”等多件事串成一个可组合的调用链。



这就需要Harness在更上层承担工具调度的责任：根据任务类型自动选择最优工具、处理工具调用的依赖关系、管理并行调用和重试策略。工具链调度已经在实际系统中被广泛采用，例如根据task.type自动匹配数据库查询工具或文件处理工具。



\### 2.6 工具的边界：并非万能



工具调用虽然强大，但也有其边界。首先，模型可能生成无效参数，导致工具执行失败；其次，工具的执行结果需要被模型正确理解和利用——如果工具返回的结果过于冗长或结构复杂，模型可能无法有效提取关键信息；最后，工具调用存在安全隐患，恶意的工具调用请求可能导致系统被攻击。



这些问题恰恰说明了Tool之上还需要Skill的抽象，以及Harness的统一治理。





\## 第三章：Skill——可复用的能力单元



\### 3.1 Skill的崛起：从“拼模型”到“拼生态”



2026年的AI竞争格局发生了一个根本性的转变。头部模型之间的性能差距在快速缩小，模型本身正在变得像电力一样普及和同质化。新的分水岭出现了：你的Agent有没有一个足够丰富的Skill层？



Cursor去年的爆火，本质不是它的模型多强（底层也是Claude和GPT），而是它的Skill做得足够深——能够理解当前光标位置、能解析整个项目上下文、能规划多步文件操作。OpenClaw的成功也不是因为使用了某个独家模型，而是把鼠标移动、点击、滚动、截图分析等操作打包成了可组合的Skill库。



模型的通用能力已经不再是护城河。谁能在模型之上构建更厚、更稳、更适配业务的Skill层，谁才有真正的议价权。



Anthropic对Skill的重视程度可以从其招聘方向看出来——超过一半的岗位在招“Agent Tooling Engineer”；OpenAI的GPT Store也早已改名为“Actions + Skills”。



\### 3.2 Skill的明确定义



Skill到底是什么？它既不是脚本，也不是插件。Skill是一个封装了 \*\*“感知-决策-执行-反馈”完整闭环\*\*的可复用能力单元。



Skill将对特定领域问题的解决经验——包括操作步骤、决策逻辑、异常处理规则、边界条件——预先封装成一个完整的“能力包”。当Agent遇到相关问题域时，它可以直接调用这个Skill，而不是每次都要从头推导解决方案。



用一个比喻来理解：如果Tool是Agent的“手脚”，可以执行具体的原子操作（如“查询数据库”“发送HTTP请求”），那么Skill就是Agent的“职业能力”——一个律师如何写合同、一个医生如何做诊断、一个程序员如何调试代码。Skill不仅包含了要调用哪些Tools，还包含了调用的顺序、判断条件、异常处理、结果验证等完整的“操作手册”。



从技术实现角度看，Skill以一个文件夹的形式存在，包含：

\- \*\*SKILL.md\*\*：特定任务的指导性说明文件，包含YAML元数据和Markdown指令

\- \*\*Scripts\*\*：可执行脚本（Python、Shell、JS等）

\- \*\*Resources\*\*：附加资源文件（模板、数据文件、参考文档等）



\### 3.3 Anthropic Agent Skills：三层渐进式披露



Anthropic在2025年底推出的Agent Skills协议，其核心设计理念是 \*\*“渐进式披露”\*\* （Progressive Disclosure）。技能的信息分为三个层次，Claude按需逐步加载，既确保必要时不遗漏细节，又避免一次性将过多内容塞入上下文。



\*\*第一层：元数据（Metadata）\*\* ——每个技能的SKILL.md文件开头都有YAML元信息，包括技能的名称和描述。在智能体启动时，Claude会预加载所有已安装技能的元数据到系统提示中。元数据提供了何时使用该技能的“线索”，但不包含具体实现细节，几乎不占用上下文窗口。



\*\*第二层：技能主体（SKILL.md内容）\*\* ——如果Claude判断某个技能与当前任务相关，它会进一步读取并加载该技能完整的SKILL.md内容作为上下文。这个Markdown文件中包含对任务的详细指令、注意事项、示例等，指导模型如何利用该技能解决问题。



\*\*第三层：附加文件和脚本\*\* ——对于更复杂或特定的场景，技能文件夹中可以包含脚本或额外说明文档。SKILL.md可以通过Markdown链接引用这些文件。当且仅当需要时，Claude才会加载或运行这些附加内容。



这种三层结构实现了动态的上下文管理：无关任务时几乎不消耗上下文窗口；一旦识别出相关技能，再逐步加载详细内容。



\### 3.4 Skill vs. Tool：核心区别



理解Skill与Tool的区别，是理解Agent能力架构的关键。



\*\*粒度不同\*\*：Tool是原子操作（如“获取当前天气”“发送一封邮件”），Skill是复合能力（如“处理客户退货请求”），Skill内部会调用多个Tool完成一个完整业务流程。



\*\*状态不同\*\*：Tool通常是无状态的，每次调用独立；Skill可以携带状态，能够跟踪任务的执行进度、中间结果和上下文。



\*\*包含知识\*\*：Tool仅提供功能接口；Skill除了功能接口，还包含了“什么时候调用什么Tool”“如何判断调用成功”“出错后如何处理”等经验和规则。



\*\*封装层级\*\*：Tool在Harness的工具链层被定义和管理；Skill在Harness的更上层，是对Tools和流程的再封装与编排。



用一句话概括：\*\*Tool教模型“怎么操作”，Skill教模型“怎么做成一件事”\*\* 。Skill是Agent从“有工具”到“有能力”的跨越。



\### 3.5 Skill生态的工程化价值



从单体Agent到可组合Skill的范式转变，带来了深远的工程化价值。



传统AI智能体采用单体架构，每个Agent需独立集成感知、决策、执行等模块，导致开发效率低、复用性差。例如，金融客服Agent和医疗诊断Agent需要各自独立开发所有的能力模块，大量重复开发且跨领域能力迁移困难。



Skill的出现改变了这一格局。通过标准化接口（MCP协议+SkiLL注册中心），Skill实现了“即插即用”，支持快速构建跨领域智能体。Agent通过组合Skills实现复杂任务——例如，一个招聘Agent可以同时加载“简历解析Skill”“面试评价Skill”“人才库匹配Skill”，而这些Skill可能来自不同团队或不同厂商。



Gartner预测，到2026年75%的AI项目将聚焦于可组合Skill而非单体Agent，标志着AI开发从“手工作坊”向“工业化组装”转型。





\## 第四章：Harness——让智能体“被放心地做”



\### 4.1 Harness的本质定义



如果说Model是Agent的大脑，Tool是手脚，Skill是职业能力，那么Harness就是Agent的 \*\*“神经系统+免疫系统+骨骼系统”\*\* ——它连接所有组件、管理系统运行、保障安全稳定。



从技术视角看，Harness是包含工具链、记忆系统、安全护栏和验证闭环的工程化框架。其核心价值在于将模型的“潜在能力”转化为“可稳定交付的业务价值”。



Harness一词原意是“马具、挽具、缰绳或安全带”，放在AI语境中，可以译为 \*\*“智能体执行约束层”\*\* 。模型决定AI“能做什么”，Harness决定AI“能否被放心地做”。



\*\*核心公式\*\*已经得到业界广泛共识：



> \*\*AI Agent = 基础模型 + Harness系统\*\*



基础模型决定“能做什么”，Harness决定“能稳定做成什么”。这种关系类似于操作系统与应用软件——再强大的CPU也需要配套的内存管理、进程调度和安全机制才能发挥作用。



\### 4.2 Harness需要解决的三大“裸模型”缺陷



为什么必须有Harness？因为裸模型存在三个天然缺陷：



\*\*无记忆\*\*：模型本身没有跨会话的持久化记忆。每次对话都是从零开始，之前的上下文和知识积累无法延续。



\*\*不能执行代码\*\*：模型只能生成代码，不能实际运行它。没有Harness的代码执行环境，模型生成的代码就是一堆无法落地的文本。



\*\*无工具调用执行能力\*\*：模型可以输出工具调用请求，但无法真正执行API调用或数据库查询。真正的执行必须在Harness层面完成。



过去行业一度聚焦于模型本身，但在实际生产中，如何系统性地使用模型被视为创造价值的关键。行业逐渐认识到，模型能力的瓶颈往往在模型之外——优化Harness可以显著提升Agent性能。



\### 4.3 三次瓶颈迁移：从Prompt到Context到Harness



AI工程化能力的发展经历了三个典型阶段，每个阶段都伴随着核心瓶颈的外移：



\*\*第一阶段：Prompt Engineering（2022-2024）\*\* ——解决“如何有效表达需求”的问题。核心挑战是如何通过指令优化、角色扮演等技巧弥补模型理解能力的不足。典型案例包括用“分步思考”提示提升推理准确率。



\*\*第二阶段：Context Engineering（2025）\*\* ——解决“如何提供有效信息”的问题。通过上下文窗口扩展、检索增强生成（RAG）等技术，构建模型的知识底座。某主流云服务商的测试显示，优化后的上下文管理可使任务成功率提升40%。



\*\*第三阶段：Harness Engineering（2026至今）\*\* ——当前阶段的核心矛盾是：即使模型能理解需求、获取知识，仍会因缺乏执行环境而失败。典型场景包括：代码生成工具因缺少依赖管理而报错；数据分析Agent因权限不足无法访问数据库；客服机器人因缺乏情感识别能力激化用户矛盾。



每一次瓶颈迁移，都标志着工程化能力的升级——从“如何问得好”到“如何给得好”再到“如何执行得好”。



\### 4.4 Harness的四层架构



一个完整的Harness系统通常包含四大核心模块：



\*\*工具链集成层\*\*——提供模型与外部系统的交互接口。包括标准化API适配器（如将数据库操作转换为自然语言指令）、异构系统连接器（支持对接ERP、CRM等业务系统）、动态工具调度（根据任务类型自动选择最优工具）。这一层是Harness的基础，负责连接模型与真实世界。



\*\*记忆管理系统\*\*——构建Agent的长期记忆能力。包括任务状态跟踪（记录多轮对话中的上下文）、知识图谱更新（将新获取的信息结构化存储）、经验回放机制（从历史成功/失败案例中学习）。记忆系统解决了模型的“连续性错配”问题，让Agent能够在长周期任务中保持状态一致性。



\*\*安全护栏体系\*\*——防止模型输出有害内容或越权操作。包含输入过滤（检测恶意指令）、输出校验（验证生成内容的合规性）、权限控制（基于RBAC的细粒度访问管理）。在AI从“内容生成者”变成“任务执行者”后，风险边界发生了根本变化——传统大模型出错主要是“说错话”，AI Agent出错可能是“做错事”，安全护栏的重要性不言而喻。



\*\*验证闭环机制\*\*——通过自动化测试和反馈优化持续提升可靠性。包括单元测试框架（验证单个工具的正确性）、集成测试环境（模拟真实业务场景）、持续学习管道（将生产环境数据反哺模型训练）。



\### 4.5 Harness为什么成为新的竞争焦点



2026年以来，AI Agent的产业焦点正在从模型能力转向执行能力。真正决定Agent能否落地的，已经不只是大模型本身，而是围绕模型构建的Harness。



近期国内外科技公司的动向已经清晰地显示了这一趋势：



\*\*OpenAI\*\*在2026年发布关于Harness Engineering的文章时指出，随着Codex等智能体承担越来越多软件生命周期工作，真正困难的问题正在转向如何设计环境、反馈循环和控制系统。软件工程纪律并没有消失，只是更多体现在工具、抽象和反馈回路等“脚手架”之中。



\*\*Google Cloud\*\*推出Gemini Enterprise Agent Platform，将模型选择、Agent构建、集成、DevOps、编排和安全能力整合到统一体系中。Agent Runtime、Agent Identity、Agent Registry、Agent Gateway等能力的设计，已经非常接近“智能体执行约束层”的工程形态。



\*\*Microsoft\*\*提出Zero Trust for AI，强调零信任原则需要扩展到AI全生命周期，包括数据、模型、部署和智能体行为。随着组织采用自主和半自主AI Agent，过度授权、被操纵或目标偏离的智能体可能带来新的风险。



巨头们不约而同地押注Harness竞争，标志着AI产业从“比模型大”进入“比控制好”的新阶段。





\## 第五章：四大支柱的协同——外部化的统一视角



\### 5.1 从“内部记忆”到“外部认知基础设施”



上海交大综述论文将记忆、技能、协议和Harness工程统称为LLM Agent的\*\*四大支柱\*\*，并用 \*\*“外部化”\*\* 这一概念将它们串联起来——技术进步不再依靠模型学会更多，而是将能力以合适的形式放在模型之外，让模型用现有的推理能力就能可靠地调用它们。



这一理念借鉴了认知科学家Don Norman的“认知工件”理论。认知工件的强大之处不在于它扩展了人类的能力，而在于它\*\*重新组织了问题的形式\*\*，让主体用已有的能力就能更可靠地解决问题：



\- 购物清单不是扩展了人的记忆容量，而是把“回忆”问题变成了“识别”问题；

\- 地图不是让人导航变得更强，而是把空间关系从隐性变成可见。



同样的逻辑正在LLM Agent上发生。外部工件的力量在于表征变换——它重新组织了问题的形式，让模型用现有的推理能力就能更可靠地解决原本困难的问题。



\### 5.2 四种外部化的分工



在外部化的统一框架下，四大支柱各司其职：



\*\*Memory（记忆）外部化状态\*\*：将跨时间的状态信息从模型的有限上下文窗口转移到外部的持久化存储中。模型不再需要“记住”所有东西，只需要知道“去哪里找”。



\*\*Skills（技能）外部化程序性知识\*\*：将领域专家的操作流程、决策规则、异常处理经验封装成可复用的能力模块。模型不再需要“学会”如何做一件事，只需要知道“调用哪个Skill”。



\*\*Protocols（协议）外部化交互结构\*\*：将Agent与外部世界的交互方式标准化，降低临时约定带来的协调性脆弱性。模型不再需要“理解”每个工具的内部细节，只需要遵循标准化的接口格式。



\*\*Harness Engineering外部化协调与治理\*\*：作为统一层，负责将上述三种外部化组件协调到一个受控的执行框架中。Harness提供了工具链集成、记忆管理、安全护栏、验证闭环、编排调度等工程化支撑，确保整个系统稳定可靠地运行。



\### 5.3 一个完整Agent的运行闭环



结合以上四个概念，一个完整的AI Agent运行闭环可以表述为：



> \*\*感知 → 决策 → 执行 → 沉淀\*\*



\*\*感知阶段\*\*：模型接收用户输入和Harness编排的上下文（包括从记忆系统检索的相关历史信息）。



\*\*决策阶段\*\*：模型进行推理分析，判断是否需要调用工具，或者直接生成最终回答。如果需要调用工具，模型输出结构化的工具调用请求。



\*\*执行阶段\*\*：Harness获取工具调用请求，通过工具链集成层调用相应的Tool或Skill，执行实际的操作（API调用、数据库查询、代码执行等）。



\*\*沉淀阶段\*\*：执行结果返回模型继续推理，同时重要的经验、成功/失败模式被Skill系统捕获和沉淀，供后续任务复用。



这个闭环持续迭代，直到任务完成或模型决定终止。模型的输出、工具的调用链、执行的反馈都通过Harness被记录和审计，构成了完整的可追溯性。



值得注意的是，Harness在这一闭环中扮演着“闭环系统”的角色——它不仅是连接的桥梁，更是持续优化的引擎。最新一代的Agent框架创新性地将Harness Engineering思想引入AI领域，通过构建“感知-决策-执行-沉淀”的闭环系统，实现了Agent能力的指数级增长。





\## 第六章：Harness工程的产业实践与未来展望



\### 6.1 主流Harness框架选型



2026年的AI Agent工程化领域，已经涌现出多种Harness框架实现。每一类框架在架构设计、编排能力、工具集成、记忆管理、安全机制和可观测性等维度各有侧重。



\*\*LangChain + LangGraph\*\*是目前生态最成熟的方案。LangChain提供了从提示词模板、工具调用、记忆管理到链式调用的全套组件，而LangGraph在此基础上引入了图结构的状态管理能力，可以构建更复杂的、支持分支和循环的Agent工作流。其LCEL（LangChain Expression Language）声明式语法大幅降低了Agent编排的复杂度。



\*\*AutoGPT\*\*则以自主性著称。它构建了“思考-计划-执行-观察”的自主循环，Agent可以自己决定下一步做什么、调用哪些工具、如何验证结果。这种设计非常适合长期运行的自主任务，但同时也带来了可控性挑战。



\*\*AutoGen（Microsoft）\*\* 走的是多Agent协作路线。它允许多个Agent以对话形式协作完成任务——一个Agent负责规划、一个Agent负责执行、一个Agent负责审核。这种“分而治之”的架构在复杂任务场景中表现优异。



\*\*CrewAI\*\*则强调角色化的多Agent团队。每个Agent被分配特定角色（如研究员、分析师、写作者），Agent之间通过预定义的角色关系进行协作，模拟真实团队的工作方式。



选型的关键不在于哪个框架更“强”，而在于你的场景需要什么样的Harness。单Agent快速原型用LangChain，多Agent复杂协作用AutoGen或CrewAI，长期自主运行用AutoGPT。



\### 6.2 工程实践中的关键考量



在真实的Harness工程落地中，以下几个问题常常决定成败：



\*\*上下文管理\*\*是Harness工程中最容易被低估的技术点。Harness负责为模型配置上下文，但如何让信息在有限的上下文窗口内高效组织，如何通过检索机制补充外部知识，如何压缩冗余信息节省token——这些问题需要精心设计的策略。



\*\*安全护栏\*\*在AI从“说”到“做”的过程中变得至关重要。Harness需要设置输入过滤、输出校验、权限控制、审计日志等多层防线。特别是权限控制，必须基于最小权限原则，确保Agent只能访问必要的资源。



\*\*可观测性\*\*是Harness工程区别于传统提示词工程的核心差异。对Agent而言，你需要追踪的是“模型输出了什么请求”“工具执行了什么操作”“每一步用了多长时间”“哪里出错了”。Harness提供的可观测性基础设施直接决定了系统的可调试性和可运维性。



\*\*版本演进\*\*是Harness工程面临的长期挑战。随着Agent的能力迭代，Skills的版本更新、工具的接口变更、模型的性能变化都可能破坏现有的工作流。Harness需要提供版本管理和兼容性策略，确保系统在演进过程中的稳定性。



\### 6.3 模型决定上限，Harness决定下限



在Harness工程领域，有一条被反复引用的经验法则：



> 模型决定上限，Harness决定下限。



模型越强，Agent的“天花板”越高；但真正决定一个Agent系统在生产环境中是否稳定可靠、是否安全可控的，是Harness的质量。



LangChain的实验提供了有力佐证：不改模型，只改Harness，就把准确率从52.8%提到66.5%。这个13.7个百分点的提升，不是来自更聪明的模型，而是来自更聪明的使用模型的方式——上下文怎么给、约束怎么定、出错怎么反馈。



这条法则揭示了Agent开发的一个核心思维转变：\*\*与其等待下一个更强的模型，不如思考如何更好地使用现有的模型\*\*。而“更好地使用”，就是Harness工程的全部意义。



\### 6.4 挑战与开放问题



尽管Harness工程的价值已被广泛认可，但这一领域仍面临诸多挑战和开放问题：



\*\*评估与治理\*\*——如何评估一个Agent系统的Harness质量？目前还没有成熟的量化指标和方法论。传统的基准测试（如MMLU、HumanEval）主要针对模型本身，而对Agent的完整执行系统缺乏有效评估手段。



\*\*参数化能力vs外部化能力的权衡\*\*——哪些能力应该由模型本身学习（参数化），哪些能力应该由Harness外部化实现？这个权衡没有标准答案，取决于具体的任务场景和资源约束。



\*\*自我进化的Harness\*\*——最前沿的方向是让Harness本身具备自我进化的能力。通过从Agent的执行日志中学习和优化，Harness可以自动调整工具调用策略、优化上下文管理、完善安全规则。这种自我进化的Harness将把Agent系统从“人工设计”推向“自动优化”的新阶段。



\*\*长期共演\*\*——模型和Harness如何长期协同演进？模型的能力在快速提升，Harness的设计如何适应这种变化？这不仅是技术问题，也是组织和生态系统的问题。





\## 结语：从“模型中心”到“系统中心”



回顾2022年到2026年的AI Agent发展历程，一条清晰的能力外移轨迹贯穿始终：



阶段一，能力几乎全部封装在模型的参数中。阶段二，部分能力被迁移到上下文层，通过Prompt工程和RAG实现。阶段三，能力的重心进一步外移到Harness层——记忆、技能、协议、工具链、安全护栏、验证闭环，所有这些“模型之外”的组件构成了智能体可靠运行的基础设施。



这一演进的本质，是从 \*\*“模型中心”\*\* 走向 \*\*“系统中心”\*\* 。模型依然重要，但它不再是唯一的关注焦点。真正决定Agent成败的，是模型、工具、技能和Harness四者构成的完整系统——模型提供认知核心，工具提供操作接口，技能封装可复用能力，Harness负责统一协调与治理。



对于今天的AI从业者而言，理解Agent = Model + Harness这个公式，意味着不仅仅是去关注最新的模型突破，更要系统性地思考：



你的Agent有足够的工具来执行任务吗？你的Skill层是否封装了业务的领域知识并可供复用？你的Harness能否在保证安全的前提下让Agent被稳定、可控地执行任务？



这些问题，恰恰是AI Agent从“有趣的Demo”走向“可靠的生产系统”所必须回答的。正如上海交大综述论文所指出的那样：Agent的实际进展，越来越取决于模型之外的外部认知基础设施，而非模型本身的能力提升。



而这场从模型到系统的思维转变，才刚刚开始。
