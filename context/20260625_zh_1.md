## 第一篇：为什么是 Rust？—— 环境搭建与所有权初探

### 一、为什么 2026 年还要学 Rust？

在回答“怎么学”之前，先回答“为什么学”。

Rust 连续多年在 Stack Overflow 开发者调查中蝉联“最受开发者喜爱的编程语言”。这不仅仅是社区的自嗨——字节跳动、阿里巴巴、Cloudflare 等公司正在将核心系统向 Rust 迁移，覆盖数据库、网络代理、AI 终端工具等底层组件。Kubernetes 生态中，Krustlet、Linkerd 和 TiKV 等知名项目全部使用 Rust 编写。

但“流行”不是学习的理由。Rust 真正打动系统程序员的，是它解决了编程语言领域一个长期存在的困境：**性能与安全不可兼得**。

C/C++ 给你极致的性能和底层控制力，但手动内存管理带来了无穷无尽的悬垂指针、缓冲区溢出、数据竞争——2022 年公开披露的 CVE 漏洞中，62% 与内存管理缺陷直接相关。Java/Go/Python 给你内存安全（通过 GC 或运行时检查），但牺牲了性能和可预测性。

Rust 的选择是：**在编译期完成内存安全检查，运行时零开销**。它通过所有权系统、借用检查器和生命周期标注，在编译时构建起完整的内存安全防线，同时通过零成本抽象实现与 C/C++ 相当的性能。某云厂商将存储服务从 C++ 重构为 Rust 后，系统崩溃率下降 92%，吞吐量提升 35%。

Rust 不仅仅是一门新语言——它是一种**新的编程思维**。理解 Rust 的所有权模型，会让你对内存管理、并发安全有更深刻的认识，这种认识可以迁移到其他语言中。

### 二、环境搭建：Rustup + Cargo

Rust 的工具链体验在主流语言中属于第一梯队。安装 Rust 的主要方式是通过 `rustup`——它既是安装器，也是版本管理工具。

**安装（macOS/Linux/其他类 Unix）：**

```bash
curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs | sh
```

**安装（Windows）：** 下载 `rustup-init.exe` 并运行，需要 Visual Studio 2013 及以上版本的 C++ 构建工具。

安装完成后，验证：

```bash
rustc --version   # 编译器版本
cargo --version   # 包管理工具版本
```

**国内用户**建议配置镜像源以加速 crate 下载。在 `~/.cargo/config.toml` 中添加：

```toml
[source.crates-io]
replace-with = 'ustc'

[source.ustc]
registry = "https://mirrors.ustc.edu.cn/crates.io-index"
```

### 三、第一个程序：Hello World

先用 `cargo` 创建第一个项目：

```bash
cargo new hello_rust --bin
cd hello_rust
```

`cargo new` 会生成如下结构：

```
hello_rust/
├── Cargo.toml          # 项目配置文件（类似 package.json / go.mod）
└── src/
    └── main.rs         # 源代码入口
```

查看 `src/main.rs`：

```rust
fn main() {
    println!("Hello, world!");
}
```

编译运行：

```bash
cargo run   # 调试模式编译并运行
cargo build --release  # 发布模式（启用优化）
```

`println!` 中的 `!` 表示这是一个**宏（macro）**，而非普通函数。Rust 的宏系统非常强大，后续会深入。

### 四、变量与可变性

在 Rust 中，变量默认是**不可变（immutable）**的。这是 Rust 设计哲学的重要体现：默认安全，需要可变时显式声明。

```rust
fn main() {
    let x = 5;        // 不可变绑定
    // x = 6;         // 编译错误！cannot assign twice to immutable variable
    
    let mut y = 10;   // 可变绑定，使用 mut 关键字
    y = 11;           // OK
    println!("x = {}, y = {}", x, y);
}
```

这一设计看似简单，却深刻影响了代码风格：它迫使程序员显式区分哪些数据会变化，哪些不会，从而减少隐式状态变更带来的 bug。

Rust 还支持**变量遮蔽（shadowing）**：

```rust
fn main() {
    let x = 5;
    let x = x + 1;    // 新变量遮蔽了旧的 x
    {
        let x = x * 2; // 在内部作用域中再次遮蔽
        println!("内部: {}", x); // 12
    }
    println!("外部: {}", x); // 6
}
```

遮蔽与 `mut` 不同：遮蔽创建了一个全新的变量，可以改变类型；`mut` 只能改变同一类型的值。

### 五、所有权（Ownership）—— Rust 最核心的概念

> 所有权是 Rust 最独特、也最让初学者感到困惑的特性。但它也是理解 Rust 的钥匙。一旦理解了所有权，Rust 的其他特性都会变得豁然开朗。

**所有权的三条核心规则**：

1. **Rust 中的每一个值都有一个所有者（owner）。**
2. **同一时刻，一个值只能有一个所有者。**
3. **当所有者离开作用域时，该值会被自动释放（drop）。**

#### 5.1 作用域与自动释放

```rust
fn main() {
    {                      // s 尚未声明
        let s = String::from("hello");  // s 进入作用域
        // 可以使用 s
    }                      // s 离开作用域，内存被自动释放
    // 此处不能再使用 s
}
```

`String::from` 在堆上分配内存，但 Rust 不需要你手动调用 `free`——当 `s` 离开作用域时，Rust 自动调用 `drop` 函数释放内存。这类似于 C++ 的 RAII（资源获取即初始化），但由编译器强制执行。

#### 5.2 移动（Move）：所有权的转移

对于**存储在堆上**的数据类型（如 `String`、`Vec`），赋值操作会**转移所有权**，而非拷贝数据：

```rust
fn main() {
    let s1 = String::from("hello");
    let s2 = s1;  // s1 的所有权被移动到 s2
    
    // println!("{}", s1);  // 编译错误！s1 已失效
    println!("{}", s2);     // OK
}
```

这一行为被称为**移动（move）**。它的内存模型是：

- `s1` 持有指向堆内存的指针、长度、容量
- `let s2 = s1` 将这三个字段**复制**给 `s2`，但**不复制**堆上的数据
- 然后 Rust 认为 `s1` 已失效，防止出现两个指针指向同一块内存的双重释放问题

这是 Rust 在不使用 GC 的前提下保证内存安全的关键机制之一。

#### 5.3 克隆（Clone）：深拷贝

如果确实需要复制堆上的数据，可以使用 `.clone()`：

```rust
fn main() {
    let s1 = String::from("hello");
    let s2 = s1.clone();  // 深拷贝堆数据
    println!("s1 = {}, s2 = {}", s1, s2);  // 两者都有效
}
```

`.clone()` 会产生运行时开销，应谨慎使用。

#### 5.4 拷贝（Copy）：栈上数据的复制

对于**存储在栈上**的简单类型（如整数、布尔值），赋值默认是**拷贝**而非移动：

```rust
fn main() {
    let x = 5;
    let y = x;    // x 是整数类型（实现了 Copy trait），这里是拷贝
    println!("x = {}, y = {}", x, y);  // 两者都有效
}
```

这些类型实现了 `Copy` trait，它们在赋值后仍然有效。常见的 `Copy` 类型包括：所有整数类型、布尔类型、浮点数类型、字符类型，以及由这些类型组成的元组。

#### 5.5 所有权与函数

将值传递给函数时，同样会发生所有权转移或拷贝：

```rust
fn main() {
    let s = String::from("hello");
    take_ownership(s);     // s 的所有权被移动到函数中
    // println!("{}", s);  // 编译错误！s 已失效
    
    let x = 5;
    makes_copy(x);         // x 实现了 Copy，这里发生拷贝
    println!("{}", x);     // OK
}

fn take_ownership(some_string: String) {
    println!("{}", some_string);
}  // some_string 离开作用域，内存被释放

fn makes_copy(some_integer: i32) {
    println!("{}", some_integer);
}
```

函数返回值也会转移所有权：

```rust
fn main() {
    let s1 = gives_ownership();         // 返回值移动给 s1
    let s2 = String::from("hello");
    let s3 = takes_and_gives_back(s2);  // s2 被移动进函数，返回值移给 s3
    // s2 已失效
}

fn gives_ownership() -> String {
    let some_string = String::from("hello");
    some_string  // 所有权移出
}

fn takes_and_gives_back(a_string: String) -> String {
    a_string  // 所有权移入，再移出
}
```

#### 5.6 所有权总结

所有权的设计解决了两个核心问题：

1. **内存安全**：每个值只有一个所有者，所有者离开作用域时自动释放内存，杜绝了悬垂指针和双重释放。
2. **数据竞争预防**：所有权规则（配合借用规则）在编译期排除了数据竞争的可能性。

可以将所有权理解为**一种编译期执行的资源管理协议**——它把传统上由程序员手动管理（C/C++）或由 GC 自动管理（Java/Go）的内存生命周期，变成了由类型系统静态检查的规则。

### 六、实战：一个简单的文件读取函数

让我们综合运用所有权知识，写一个读取文件内容的函数：

```rust
use std::fs::File;
use std::io::Read;

fn read_file_contents(path: &str) -> std::io::Result<String> {
    let mut content = String::new();
    let mut file = File::open(path)?;      // ? 操作符简化错误处理
    file.read_to_string(&mut content)?;    // 通过可变引用修改 content
    Ok(content)  // content 的所有权转移给调用者
}

fn main() -> std::io::Result<()> {
    let content = read_file_contents("hello.txt")?;
    println!("文件内容: {}", content);
    // content 在这里离开作用域，内存自动释放
    Ok(())
}
```

注意几个关键点：
- `read_file_contents` 的参数是 `&str`（字符串切片），这是一个**借用**，不获取所有权
- `content` 在函数内部创建，通过 `Ok(content)` 将所有权**转移**给调用者
- `read_to_string(&mut content)` 使用**可变借用**来修改 `content`，而不获取所有权

这里涉及了“借用”的概念——这正是第二篇的主题。
