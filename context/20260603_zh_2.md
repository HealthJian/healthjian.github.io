# 本科毕设成果综述：基于大模型的智能简历生成助手

> **摘要**：本文是对一项本科毕业设计工作的成果回顾，主题为**基于大模型的智能简历生成助手**——从多源简历数据工程、领域微调、岗位语义对齐，到可部署的 B/S 应用系统。全文以**技术复盘**的口吻撰写：说明解决了什么问题、采用了怎样的流水线、实验与系统验证说明了什么，以及个人在其中的工程收获。**文中已对姓名、学号、院校、导师等身份信息做脱敏处理，不涉及可识别个人的隐私数据。**

## 一、为什么要做这件事

高校毕业生规模持续扩大，简历往往是求职链路的「第一张筛选器」。现实里常见三类困境：

- **信息组织混乱**：经历素材多，却难以在有限篇幅内突出与目标岗位相关的硬技能与量化成果；
- **岗位适配不足**：同一份简历投不同 JD，缺少针对性的关键词与叙事结构；
- **格式与 ATS 不友好**：表格化排版、关键词堆砌或结构不规范，可能在申请人追踪系统（ATS）阶段被误伤。

大模型擅长润色段落，但简历生成不是普通的「续写」任务。它同时带有**强事实约束**（不能凭空编造项目）、**强结构约束**（模块、时序、术语规范）和**高风险后果**（隐私、公平性、可信度）。因此课题目标被收敛为：构建一套**端到端、可追溯、可约束**的系统——在数据层做脱敏与质量门禁，在模型层抑制幻觉并做领域适配，在应用层提供 ATS 反馈与人工可编辑闭环。

## 二、总体技术路线：四段式生成 + 三层工程

论文将核心流程概括为 **「输入解析 → 信息补全 → 岗位对齐 → 风格生成」**，并在工程上拆为三条并行主线：

```mermaid
flowchart LR
    subgraph Data["数据工程"]
        D1["多源解析"]
        D2["清洗去重"]
        D3["PII 脱敏"]
        D4["ShareGPT 指令集"]
    end
    subgraph Model["模型层"]
        M1["LoRA 领域微调"]
        M2["RAG 检索增强"]
        M3["动态 Prompt"]
    end
    subgraph App["应用系统"]
        A1["九步信息采集"]
        A2["画布编辑 + AI 润色"]
        A3["ATS 评分反馈"]
        A4["多格式导出"]
    end
    Data --> Model --> App
```

### 2.1 数据工程：从异构语料到可训练指令

研究首先建设面向**多源异构简历**的数据工程流水线，关键环节包括：

| 环节 | 做法 | 目的 |
|------|------|------|
| 编码与字段统一 | Canonical Schema 映射 JSON / 纯文本简历 | 打通结构化与非结构化来源 |
| 智能去重 | 相似样本合并 | 降低微调重复偏置 |
| 隐私脱敏 | 姓名哈希假名、电话/邮箱/证件号规则脱敏 | 满足训练与评测合规 |
| 质量分层 | 人工质检 + 门禁规则 | 控制噪声样本比例 |
| 指令构造 | 多模板 ShareGPT 多轮对话 | 兼容 LLaMA-Factory 微调 |

最终形成约 **15 万条** ShareGPT 格式 SFT 指令样本（含数据增强），并另建**独立于训练集**的 RAG 语料库，供生成阶段引用优秀简历片段。论文还设计了 **JD-简历匹配评分模型**，融合技能匹配度、经验相关性与语义相似性，为检索与 Prompt 注入提供对齐信号。

### 2.2 模型层：Qwen 基座 + LoRA + RAG + 动态 Prompt

- **基座模型**：Qwen3.5-9B-Instruct（参数高效微调场景下的开源指令模型）。
- **微调框架**：LLaMA-Factory；LoRA 配置典型为 rank=16、alpha=32，覆盖 Attention 与 FFN 线性层，FP16 + 梯度检查点，在 24GB 级 GPU 上可完成训练。
- **生成增强**：Top-K 相似简历片段 RAG 召回 + 按岗位方向、语言风格、JD 关键词注入的**动态 Prompt**。
- **岗位语义**：BERT-Whitening 对白化后的句向量做 JD-简历匹配，论文报告 Top-5 匹配准确率约 **85%**，优于 Sentence-BERT CLS 与平均池化基线。

### 2.3 应用系统：React + FastAPI 的闭环产品

系统采用**前后端分离**：前端 React 19 + TypeScript + Vite，后端 FastAPI + Pydantic；支持中英文界面。功能闭环覆盖论文所述六项能力：

1. **多源信息抽取**：教育、工作、项目、技能等字段的结构化解析；
2. **岗位导向生成**：结合求职方向或目标 JD 强化专业表述与量化；
3. **润色与局部改写**：单节 Refine、全篇 Review、按指令 Revise；
4. **ATS 评分与优化建议**：关键词、技能覆盖、经验相关、格式、量化成果五维加权；
5. **多格式导出**：Markdown / PDF / Word / PNG 等，画布实时预览；
6. **安全合规**：PII 脱敏、敏感词与偏见约束、提示注入防护。

## 三、实验与验证：数字背后在验证什么

### 3.1 消融实验：模块不是「堆料」，而是可度量增益

在相同测试集上逐步叠加模块（论文表 4-4 量级结果）：

| 变体 | BLEU-4 | ROUGE-L | BertScore | ATS 评分 |
|------|--------|---------|-----------|----------|
| 基座零样本 | 28.2 | 26.4 | 0.812 | 67.3 |
| + LoRA | 38.7 | 37.9 | 0.865 | 78.5 |
| + LoRA + RAG | 41.4 | 40.3 | 0.883 | 82.1 |
| **完整（+ 动态 Prompt）** | **43.1** | **42.6** | **0.901** | **85.6** |

解读方式：**LoRA 领域适配**带来格式与术语的最大跃迁；**RAG** 提升信息丰富度与语义相关；**动态 Prompt** 进一步拉高岗位关键词匹配等 ATS 子项。论文对四组变体做了 ANOVA 与 Bonferroni 校正的事后检验，组间差异在主要指标上达到显著水平。

与 GPT-4o、Claude-3.5-Sonnet 等闭源基线对比时，实验强调**统一输入模板、统一 Prompt、统一采样参数与后处理**，完整方法在 BLEU-4 / ROUGE-L / BertScore 上达到或超过强基线水平，同时困惑度更低，说明领域微调改善了流畅度而非单纯「更长」。

### 3.2 事实一致性：比「写得像」更重要

论文采用 NLI 冲突检测、时间规则检测与人工事实审核三重机制。完整模型相对基座：NLI 冲突率约从 14.5% 降至 **3.6%**，人工事实错误率约从 11.2% 降至 **2.8%**。这与「简历不能 hallucinate 职级与项目」的产品要求一致——**可控生成**优先于华丽辞藻。

### 3.3 系统级验证：从字段抽取到用户体验

除生成指标外，论文在独立测试环境验证模块与端到端表现，要点包括：

- **信息抽取**：教育/工作/项目/技能等字段宏平均 F1 约 **0.90**；
- **ATS 评分**：与多智能体模拟「邀请率」代理指标 Pearson **r ≈ 0.78**；高分段与低分段邀请率差异显著；
- **性能**：RTX 4090 环境端到端平均耗时约 **1.9s**（<3s 指标），A100 约 **1.5s**；推理占主要耗时，RAG 检索为毫秒级；
- **小样本用户调研**：九步采集流程、RAG 参考与 ATS 反馈获较高满意度，「生成—评估—优化」闭环被用户频繁使用。

### 3.4 安全可信：纵深防御不是附录装饰

系统实现**采集—传输—生成—存储**四层 PII 脱敏，输出侧二次扫描；公平性上结合 Prompt 约束、AC 自动机敏感词、语义偏见分类与孪生简历成对测试；提示注入在网关、应用 Schema 与输出格式校验三级拦截。论文报告常见注入场景防御成功率约 **80.7%**，平均拦截延迟 <15ms——说明**就业场景落地**时，安全与体验必须一起设计。

## 四、部署与工程化：让模型「能跑起来」

训练产物通过 **LoRA 合并 + FP16 / INT8 / INT4(AWQ)** 多精度导出，适配不同显存环境。线上推理支持多种模式：

- 开发态：PEFT 动态挂载 LoRA 适配器（约 80MB 级增量）；
- 联调态：LLaMA-Factory OpenAI 兼容 API；
- 生产态：**vLLM** + PagedAttention，量化后显存可降至约 5GB 量级，利于并发。

后端另提供 **Docker Compose** 编排（API 服务 + Redis 会话），与第 5 章容器化部署方案一致。对个人而言，这条链路把「jupyter 里能生成」推进到「服务化、可降级、可监控」——与此前在本站写的 [数据工程](moban_new_md.html?md=../../context/20260423_zh_3.md)、[RAG](moban_new_md.html?md=../../context/20260420_zh.md) 文章形成了纵向衔接。

## 五、研究贡献与个人收获（工作综述视角）

从**成果**角度，本课题的主要输出可归纳为：

| 维度 | 产出 |
|------|------|
| 数据 | 多源简历数据工程流水线 + ~15 万指令微调集 + RAG 语料 |
| 方法 | JD-简历匹配（BERT-Whitening）+ LoRA/RAG/动态 Prompt 可控生成 |
| 系统 | 可交互简历助手（抽取、生成、ATS、导出、安全合规） |
| 验证 | 自动指标、消融、事实一致性、系统性能与用户调研 |

从**个人工程成长**角度，几项体会比分数更重要：

1. **垂直 LLM 应用的第一性原理是约束**：简历场景里，interrupt 等价物是「事实锚定 + 人工画布编辑 + ATS 可解释反馈」，而不是一味提高 temperature。
2. **数据工程占用时间往往超过调参**：脱敏、去重、模板池与质量门禁决定了微调上限；没有干净指令集，LoRA 只会学会「像简历的废话」。
3. **评测要分两层**：BLEU/ROUGE 看相似度，ATS 与事实一致性看「能不能用」；就业场景必须两者同时看。
4. **AI 工具是辅助，核心设计需自己闭环**：论文按规定披露辅助工具用于文献编排、图示与部分脚本，但研究设计、实验分析与系统实现需独立把关——这也是后续写 [工程类博客](moban_new_md.html?md=../../context/20260603_zh_1.md) 时坚持的立场。

## 六、局限与后续方向（论文第 6 章要义）

诚实面对局限，反而让综述更可信：

- **真实投递反馈稀缺**：ATS 与 HR-Agent 模拟指标是代理，不能完全替代企业侧 A/B；
- **3D/多模态简历、视频简历**未覆盖；
- **跨语言、跨行业深层适配**仍需更多领域 LoRA 与合规评审；
- **更大基座与在线 RLHF/DPO** 可能进一步提升，但算力与数据闭环成本更高。

后续可沿 **多语言支持、实时反馈优化、隐私增强（差分隐私/遗忘学习）** 等方向延续——与论文展望一致，此处不展开为立项计划。

## 结语

这项毕业设计，本质上是把「大模型能写」推进到「大模型能帮求职者**写得对、写得贴、写得敢用**」。数据脱敏与质量分层解决「能不能训」；LoRA + RAG + 动态 Prompt 解决「像不像岗位」；ATS 与画布编辑解决「敢不敢投」；安全合规解决「能不能上线谈」。
